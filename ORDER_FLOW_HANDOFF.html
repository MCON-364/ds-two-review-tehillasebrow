Order

When order matters

Now let's ask: do we need the most recent thing first, or the oldest thing first?

  • Most recent first → stack
  • Oldest first → queue
Flow

When work moves

Let's think about work entering the system, waiting somewhere, and being handled later.

  • Work comes in
  • Work waits in a queue
  • Workers take work out
Handoff

When threads are involved

Once multiple threads touch the same data, we have to ask what is shared and how it is protected.

  • Shared queue
  • Shared counters
  • Shared summary state
Guiding question: Before we write code, let's ask: What does this problem care about — order, uniqueness, lookup, priority, summary, or safe handoff between threads?

Warm-up: what kind of problem is this?

Let's look at a few small situations. We are not coding yet. We are only choosing the right tool.

Scenario What we notice Likely tool
Undo the last edit in a document The most recent action is the first one undone. Deque used as a stack
Print jobs wait for a printer The first job submitted should print first. Queue
Many parts of a system submit log messages while worker threads process them One side produces work, another side consumes work. Producer / Consumer with a queue
Calculate min, max, count, and average from processed values We are summarizing many values. Stats object, stream summary, or thread-safe counters
Main takeaway: The structure is not decoration. The structure is part of the solution. If we choose the wrong structure, the code becomes harder than it needs to be.

1Problem: Undo / Redo History

Now let's start with a problem that is really about history.

Problem statement: A simple editor keeps track of actions. We can perform a new action, undo the most recent action, and redo the most recently undone action.

Step 1: Model the data

Let's keep the domain small. An action needs a description and maybe a timestamp.

public record EditorAction(
        String description,
        long timestamp
) { }

Step 2: Identify the operations

Before choosing code, let's name what the object needs to do.

public void perform(EditorAction action)
public Optional<EditorAction> undo()
public Optional<EditorAction> redo()
public int undoCount()
public int redoCount()

Step 3: Choose the data structure

Let's think about the order. When we undo, which action do we need?

We need the most recent action. That means this is a stack-shaped problem.

private final Deque<EditorAction> undoStack = new ArrayDeque<>();
private final Deque<EditorAction> redoStack = new ArrayDeque<>();
Undo stack

Stores actions we have performed and might undo.

Redo stack

Stores actions we have undone and might restore.

Step 4: Identify the algorithm

OperationWhat happens?Why?
perform(action) Push onto undo stack and clear redo stack. A new action changes history, so old redo choices no longer make sense.
undo() Pop from undo stack and push onto redo stack. The action is no longer active, but we may want to redo it.
redo() Pop from redo stack and push onto undo stack. The action becomes active again.

Step 5: Implement together

Now we'll create the class. Notice where Optional belongs: undo and redo may not have anything to return.

    private final Deque<EditorAction> redoStack = new ArrayDeque<>();

    public void perform(EditorAction action) {
        // TODO: push action onto undoStack
        // TODO: clear redoStack
    }

    public Optional<EditorAction> undo() {
        // TODO: if undoStack is empty, return Optional.empty()
        // TODO: otherwise move one action from undoStack to redoStack
    }

    public Optional<EditorAction> redo() {
        // TODO: if redoStack is empty, return Optional.empty()
        // TODO: otherwise move one action from redoStack to undoStack
    }

    public int undoCount() {

Watch for this: If perform does not clear the redo stack, the object allows redo actions from an old branch of history. That is a logic bug, not a syntax bug.

2Problem: Log Processor

Now let's make the queue problem more realistic. Work arrives from one place and gets handled somewhere else.

Problem statement: A system receives log messages. Code in different parts of the application submits messages. Worker threads process those messages and update summary counts.

Step 1: Model the data

Let's represent each message with a source, a level, a message, and a timestamp.

public enum LogLevel {
    INFO, WARN, ERROR
}

public record LogMessage(
        String source,
        LogLevel level,
        String message,
        long timestamp
) { }

Step 2: Identify the operations

Let's write the public API before the implementation. This helps us focus on behavior.

public void submit(LogMessage message)
public void start(int workerCount)
public void stop() throws InterruptedException
public int getTotalProcessed()
public Map<LogLevel, Integer> getCountsByLevel()

Step 3: Choose the data structure

This is a queue-shaped problem because messages wait to be processed in arrival order.

But now there is a twist: multiple threads are involved. A normal ArrayDeque is not enough because the queue is shared.

private final BlockingQueue<LogMessage> queue = new LinkedBlockingQueue<>();
Let's think about it: If one thread is adding messages while another thread is removing messages, what can go wrong with a plain queue?

Step 4: Identify the pattern

This is the Producer / Consumer pattern.

Producer submits message
→
Shared queue
→
Consumer thread takes message
→
Stats updated
RoleMeaning in this problemJava tool
Producer Code that calls submit Any calling thread
Queue Messages waiting to be processed BlockingQueue
Consumer Worker thread that processes messages Thread with a loop
Stats Total processed and counts by level AtomicInteger, ConcurrentHashMap

Step 5: Choose thread-safe state

Now let's list the shared state. If multiple workers update the same counters, those counters must be safe.

private final AtomicInteger totalProcessed = new AtomicInteger();
private final ConcurrentHashMap<LogLevel, AtomicInteger> countsByLevel = new ConcurrentHashMap<>();
private final List<Thread> workers = new ArrayList<>();
private volatile boolean running = false;
Why this makes sense: The queue handles safe handoff. AtomicInteger handles safe counting. ConcurrentHashMap handles safe lookup and updates by log level.

Step 6: Implement the processing loop

Now let's think about stopping. We do not want workers to stop while messages are still waiting in the queue.

while (running || !queue.isEmpty()) {
    LogMessage message = queue.poll(100, TimeUnit.MILLISECONDS);
    if (message != null) {
        process(message);
    }
}

The loop continues while the processor is running, or while there is still work left to finish.

3Problem: Calculate Stats from Processed Readings

Now let's connect this to streams and summaries. After work is processed, we often need a report.

Problem statement: A sensor system collects readings. Each reading has a sensor id, a numeric value, and a timestamp. We need count, minimum, maximum, and average.

Step 1: Model the data

public record SensorReading(
        String sensorId,
        double value,
        long timestamp
) { }

Step 2: Identify the operation

Here the input is a list, and the output is a summary. We are not changing the list. We are calculating facts about it.

public DoubleSummaryStatistics summarize(List<SensorReading> readings)

Step 3: Choose the algorithm

This is a natural stream summary problem.

return readings.stream()
        .mapToDouble(SensorReading::value)
        .summaryStatistics();

Step 4: Connect it back to threads

Now let's ask a design question. Should the processor calculate stats while messages are being processed, or should it store readings and summarize later?

Option A: summarize later

Store readings, then use streams after processing is complete.

Good when: we need flexible reports later.

Option B: update stats while processing

Update counters, totals, min, and max as each reading is processed.

Good when: we only need one known summary and want less storage.

Let's think about it: If several worker threads update min, max, total, and count at the same time, what has to be protected?

How this connects to the final

The final can change the story, but the shape of the problem will still be recognizable.

If the final says... We should think... Likely tools
“Most recent item first” This is a stack-shaped problem. Deque, push, pop, Optional
“First submitted, first processed” This is a queue-shaped problem. Queue, offer, poll, peek
“One side creates work, another side handles work” This is a producer/consumer problem. BlockingQueue, workers, safe counters
“Count, average, min, max, group by” This is a stats/report problem. streams, collectors, summary statistics
“Multiple threads update the same state” We must protect shared mutable state. AtomicInteger, ConcurrentHashMap, locks, synchronized access
Final reminder: First identify the shape. Then pick the data structure. Then name the operations. Then code. If we skip the modeling step, the code usually gets messy.
